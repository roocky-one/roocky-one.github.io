减轻软件开发中的心智负担

发表于 2019-05-08 | 更新于 2019-05-13 | 评论数： | 阅读次数：

对脏代码的零容忍、对工程细节的苛求与对完美产品的想象，我想，是开发者应有的追求。
可是，现实由于工期或团队等因素，我们不得不妥协，然后如鲠在喉，在项目后期身心俱疲。

经历几次痛苦，
要不就转而实用主义，不再继续苛求，既缓解自己，又符合公司的利益。可惜，世间就此少了个有追求的开发者。
要不就开始有心智负担，再面对细节选择，纠结到无法决定。

说的不是别人，从入行、带团队，再到创业，伙计们和我，经历好多个轮回。
下面说些通识性应对，不针对特定平台或框架，是个方法导读：

让命名，少些痛苦

为函数/变量/API的名字纠结，是我日常。

还好有CODELF，瞬间搜索GitHub所有项目的变量/函数/代码的参考命名，由于国人开发，输入中文会自动翻译成英文，很棒。
CODELF解决内部命名已充分，而对公开API的设计，还需要一些方法：

  • HTTP API 设计指南
  • Qt API Design Principles

做好随时重构的准备

再好的代码，终究会不顺眼，任其放任，彻底腐化是必然。
由此，需要灵活的重构代码，常用操作：

  • 所有文件调整统一更改命名
  • 代码自动查错、格式化
  • 全工程关联分析，函数、变量的引用随时查询、跳转
  • 多窗口代码对比
  • …

说白了，要个好的编译器/IDE，方便随时搞。过去聊这事，会引发一场圣战，不过今天，基本已是共识：

  • 用对应语言/框架/平台，主流推荐的环境(IDE)，流水不争先
  • 如无特别推荐，就用VS Code，全平台、全语言开发支持

是的，开源界对编辑器的圣战，让微软统一了。在今天，这就是最好的编辑器，极大降低开发者负担。
我除了在ssh/serial上用Vim，剩下所有事都用VS Code，包括这篇文章。

定义用户能理解的版本号

版本号本是件随意事，项目内定好规则就OK。然而今天，软件都不是孤立的，尤其是中间件，相互依赖度空前的高，版本号不再是一串UUID，还要能让用户清楚的明白兼容性等信息，保障使用者安全的使用与更新。

GitHub推广的语义化版本2.0.0，就是当今的标准。哪怕你的软件没有类似问题，还是推荐这个简单实用的版本约束，它让你的团队更省心。

良好的代码提交流程

到今天，Git优于SVN等已不再有争议，是目前版本控制事实上的标准。然而，很多团队不懂相应的开发流程，没利用到工具，还带来很多负担。

懂得Git工作流，可按照参考下面阅读。花费的时间，是值得的：

  • 《Pro Git v2》中文版
  • Git 分支 - 分支开发工作流
  • Git 工作流程

不止项目代码，任何东西都可用Git管理，包括不限于：

  • 随手的测试代码
  • 读书、学习笔记
  • 你现在看到的这篇文章
